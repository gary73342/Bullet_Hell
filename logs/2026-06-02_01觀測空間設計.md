# 開發日誌 2026-06-02_01

## 主題：Gym 包裝層 — 觀測空間設計討論

---

## 討論結論

### 1. 畫面解析度

- 原始遊戲畫面：480×640
- 決定縮放至 **84×112**（方案 B）
- 理由：保留 3:4 長寬比，避免壓扁豎版畫面導致子彈位置失真

### 2. 色彩空間

- 決定使用 **RGB 彩色**（絕對不使用灰階）
- 理由：遊戲顏色具有明確語義
  - 青色 = 我方飛機
  - 紅色 = 敵機
  - 粉色 = 敵方子彈
  - 黃色 = 我方子彈
- 灰階會導致粉色與黃色亮度相近，AI 無法區分敵我子彈

### 3. HUD 處理

- 決定使用 **Offscreen Surface**
- 遊戲物件（飛機、子彈、星空）畫在獨立的 offscreen surface，**不畫 HUD**
- 此 surface 縮放後作為 AI 的觀測值
- 顯示器畫面仍正常顯示（含 HUD），供人類觀看
- HP、分數等數值透過 `info` dict 另外傳遞
- 優點：HUD 處理與解析度縮放完全解耦

```
遊戲物件 → offscreen surface (480×640) → 縮放 → AI 觀測 (84×112, RGB)
               ↓
           display surface (480×640) → 加上 HUD → 顯示給人看
```

### 4. Frame Stacking（幀堆疊）

- 決定堆疊 **4 幀**
- 理由：單張畫面無法提供速度與方向資訊，堆疊 4 幀可讓 AI 推斷子彈軌跡
- 以 `ENEMY_BULLET_SPEED = 4`（每幀 4px）為例，4 幀可觀察到 16px 的位移，足夠判斷方向與速度
- 不使用 8 幀：記憶體與計算量翻倍，收益不顯著

### 5. Frame Skip（跳幀）

- 決定使用 **Frame Skip = 4**
- 屬於 **Gymnasium 包裝層的參數**，與 CNN 和 PPO 無關
- 機制：env.step() 每次被呼叫時，將同一個 action 執行 4 個遊戲幀，再回傳 observation
- 等效決策頻率：60 ÷ 4 = **15 次/秒**
- 4 幀堆疊涵蓋歷史：4（skip）× 4（stack）= 16 幀 ≈ 267ms
- 訓練時與 AI 實際操控時均生效

```
env.step(action)
  → 遊戲跑幀1（執行 action）
  → 遊戲跑幀2（執行 action）
  → 遊戲跑幀3（執行 action）
  → 遊戲跑幀4（執行 action）
  → 回傳 observation（最新4幀堆疊）給 CNN + PPO
```
